# 依赖与通信规则

## 何时读取

- 需要判断一个形态能依赖谁
- 需要设计跨形态交互方式
- 需要确认多个 `Store` 是否越界

## 依赖矩阵

| 主形态 | 允许依赖 |
| --- | --- |
| `Schema` | 无 |
| `Pure` | `Schema`, `Pure` |
| `Port` | `Schema` |
| `Store` | `Schema`, `Pure` |
| `Select` | `Schema`, `Pure`, `Store`（只读，可多个） |
| `Loop` | `Schema`, `Pure`, `Store` |
| `Task` | `Schema`, `Pure`, `Store`, `Port` |
| `Output` | `Schema`, `Select`, `Port` |
| `Gateway` | `Schema`, `Port` |
| `Entry` | `Schema`, `Task`, `Output`, `Select` |
| `Boot` | 全部，但仅限装配期 |

## 补充约束

- `Store` 只能通过显式入口更新
- `Select` 只能读，不得写
- `Output` 不直接依赖 `Store` 内部结构，优先依赖 `Select`
- 上层依赖 `Port`，不依赖具体 `Gateway`
- `Entry` 不得直接接触 `Store` 内部实现细节和宿主不可见对象

## 正式通信形态

正式通信形态约束的是高风险“边”。这些边必须有可机械识别的物理存在，例如宿主注册点、回调签名、`update/fixedUpdate/tick` 入口、`async/await` 语法、`Store` 的显式写入口、已类型化的事实载荷。

### `Callback`

- 定义：外部宿主把控制权打入项目内部的调用
- 只能进入：`Entry`
- 规则：只负责把外部输入转换为内部调用
- 禁止：直接展开有限流程、直接执行 IO、直接持有或修改领域真相
- 后续：需要异步编排时转交给 `Task`；需要改写真相时通过显式 `Write` 进入 `Store`

### `Tick`

- 定义：重复推进型的调度入口
- 只能进入：`Loop`
- 输入要求：显式表现为时间步长、快照或明确上下文
- 禁止：直接执行 IO、直接操作宿主不可见对象、在 `Tick` 路径上出现 `await`
- 后续：需要对外发效果时交给 `Output`；需要异步外部交互时转交给 `Task`

### `Await`

- 定义：有限流程中的异步等待
- 只能作为：`Task` 的主执行形态
- 规则：`Task` 通过 `Port` 请求外部能力并等待结果；`Gateway` 可以把外部机制包装成可等待接口，但不得在其中承载业务流程
- 禁止：在 `Pure`、`Store`、`Select`、`Loop` 中出现 `Await`；禁止在 `Entry` 中直接展开 `Await`

### `Write`

- 定义：对某个领域内权威真相的显式修改
- 只能写入：`Store`
- 主要发起者：`Loop`、`Task`
- 禁止：`Entry`、`Output`、`Gateway`、`Select`、`Pure` 直接改写 `Store`
- 禁止：通过共享引用、可变别名、全局单例、隐式回调副作用隐藏写入路径
- 要求：同一份事实的 `Write` 入口必须收敛到其权威 `Store`

### `Signal`

- 定义：已发生事实的单向、已类型化传播
- 含义：表示“某件事已经发生”，不是“请去做某事”
- 规则：必须使用显式类型；`Loop`、`Task`、`Store` 可以产出；`Output` 可以消费以决定效果请求
- 禁止：字符串事件名、动态字典载荷、把 `Signal` 当新的真相源、用未类型化全局事件总线作为主通信形态
- 选择原则：如果只是稳定读取当前状态，优先用 `Store / Select`，不要滥用 `Signal`

## 多领域 `Store` 规则

- 系统中可以同时存在多个领域级 `Store`
- 每个 `Store` 必须声明自己的作用域
- 同一份事实只能有一个权威来源
- 不同领域可以读取彼此公开的只读结果，但不得越界夺取真相所有权
- 跨领域写入必须通过正式入口
- 是否属于 UI 不是判断依据；只要一个交互子系统拥有独立真相，它就可以正当地拥有自己的 `Store`

## 额外说明

- 普通同步调用仍然存在，但不是本范式的主约束中心
- 普通只读访问也仍然存在，但不要把派生结果缓存后冒充新的真相源
