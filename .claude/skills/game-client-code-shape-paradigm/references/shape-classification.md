# 形态判定

## 何时读取

- 需要在实现前判断主形态
- 一个文件看起来同时像 `Store` / `Task` / `Output` / `Gateway`
- 需要决定当前代码该保留、拆分还是迁移

## 覆盖范围

本范式覆盖游戏客户端中的主要运行时代码，包括但不限于：

- 战斗、Buff、AI、生成、交互、移动、结算
- HUD、菜单、背包、任务、排行榜、地图、回放、教程
- 登录、匹配、进房、切场景、存档、恢复
- 网络、文件、平台、资源、音频、特效、动画、相机接入

不覆盖：

- 编辑器工具
- 构建脚本
- 内容生产流水线
- 服务端代码

## 判定原则

- 先判形态，再允许实现
- 形态按“实际拥有的能力”判定，不按业务名称或作者意图判定
- 一个文件或模块只能有一个主形态
- 如果同时命中两个高风险主形态，结论是拆分，不是二选一
- 命中更高优先级形态后，不允许降级归类
- 如果实现必须突破依赖矩阵，优先判断为形态判错或边界拆分不足

## 五个判定轴

按以下顺序判断，不要调换：

1. `入口来源`：它是被引擎、UI、SDK、平台、系统回调直接打入，还是由项目内部显式调用进入
2. `边界可见性`：它是否直接接触宿主不可见对象，例如引擎节点、组件、场景句柄、SDK 对象、文件句柄、socket、资源句柄
3. `执行形态`：它的主执行方式是同步计算、有限流程，还是重复推进
4. `副作用权限`：它只做计算、只决定“要发什么效果”，还是会直接执行外部交互
5. `状态权限`：它无状态、只读派生，还是持有某个作用域内的权威真相

## 闭集主形态

| 主形态 | 一句话定义 |
| --- | --- |
| `Schema` | 定义静态结构，不参与运行 |
| `Pure` | 同步、确定、无副作用的计算 |
| `Store` | 保存某个领域的权威运行时真相 |
| `Select` | 从一个或多个 `Store` 产出稳定只读结果 |
| `Loop` | 负责重复推进 |
| `Task` | 负责把一件有限的事情办完 |
| `Entry` | 承接所有从外部宿主打进来的入口 |
| `Output` | 决定是否发效果、发什么效果 |
| `Port` | 定义如何请求外部能力 |
| `Gateway` | 真正接触宿主不可见对象并实现外部能力 |
| `Boot` | 负责装配与初始化 |

## 主形态判定顺序

按优先级从高到低判定，首个命中的形态优先：

1. `Boot`：只负责装配和初始化
2. `Entry`：由引擎、UI、SDK、平台或系统回调直接打入
3. `Gateway`：直接持有或操作宿主不可见对象
4. `Loop`：负责重复推进
5. `Task`：负责有开始和结束的有限流程
6. `Output`：根据内部真相或事实决定效果请求，但自己不持有宿主对象
7. `Store`：持有某个明确领域或作用域内的权威真相
8. `Select`：只做稳定的只读派生
9. `Pure`：只做同步、确定、无副作用的计算
10. `Schema`：只定义结构

## 以能力判定，而不是按语义判定

- 只要直接接收外部回调，它就是 `Entry`，哪怕内部转发的是业务逻辑
- 只要直接摸到引擎节点、SDK 对象、文件句柄或资源句柄，它就是 `Gateway`
- 只要负责持续推进世界或子系统演化，它就是 `Loop`
- 只要负责跨多个步骤把一件事办完，它就是 `Task`
- 只要持有某个作用域内“当前真实值是什么”的最终裁决权，它就是 `Store`
- 只要只读、可重算、无独立生命周期，它才可能是 `Select`

## 常见拆分信号

- 同一个文件既接回调又展开异步流程：拆成 `Entry + Task`
- 同一个文件既决定效果又直接操作引擎对象：拆成 `Output + Gateway`
- 同一个文件既持续 tick 又发 IO：拆成 `Loop + Task` 或 `Loop + Gateway`
- 同一个文件既保存权威真相又承载复杂纯计算：拆成 `Store + Pure`
- 同一个文件把草稿态、加载态、分页态挂在“读模型”里：重判为对应领域的 `Store`

## 裁决规则

- 不能为了少拆文件而降级归类
- 不能用 `utils`、`helper`、`service`、`manager`、`common`、`misc` 逃避归类
- 命中两个高优先级形态时，结论永远是拆分
- 一旦需要非法依赖，先重判主形态，再设计新边界

## 解释规则

- ECS 是运行时组织方式，不是主形态分类方式
- MVVM 是 UI 组织方式，不是能力边界定义方式；`ViewModel` 内部仍需继续拆成 `Store`、`Select`、`Task`、`Output`
- 响应式实现手段不会自动决定形态；决定形态的仍然是谁持有真相、谁接入口、谁发副作用、谁负有限流程
