# 形态规则

## 何时读取

- 已经判定出主形态，想确认它允许做什么、禁止做什么
- 需要决定某段代码该留在当前文件，还是应拆给相邻形态
- 需要确认文件命名和落位方式

## `Schema`

- 定义：只描述结构，不参与运行
- 允许：值对象、配置、枚举、错误码、协议载荷、静态常量、与运行时无关的轻量构造辅助
- 禁止：状态、异步、IO、宿主对象、环境读取、流程、业务决策
- 判断：删掉运行时上下文后它仍然成立；一旦出现订阅、缓存、惰性计算、环境读取或状态保存，就不再是 `Schema`

## `Pure`

- 定义：同步、确定、无副作用的计算
- 允许：公式计算、校验、过滤、状态迁移函数、差异计算、归一化、决策表、纯判断
- 禁止：IO、异步、时间、随机数、环境读取、全局单例、宿主对象、隐式状态
- 判断：相同输入必须得到相同输出；只要结果依赖外部环境而不是参数，就不是 `Pure`

## `Store`

- 定义：在某个明确领域或作用域内保存并维护权威运行时真相
- 允许：保存当前真实状态、提供受控读取/订阅/更新入口、维护一致性边界、版本、原子更新和局部不变式
- 禁止：IO、异步流程编排、直接接触宿主不可见对象、复杂纯计算、表现拼装、直接对外发效果
- 判断：它之所以是 `Store`，不是因为“重要”，而是因为它对某个作用域内“当前真实值是什么”拥有最终裁决权

## `Select`

- 定义：从一个或多个 `Store` 中提取、整理并输出稳定只读结果的读侧边界
- 允许：只读 `Store`、排序、分组、聚合、裁剪、映射、格式化、生成稳定读模型或消费模型
- 禁止：回写 `Store`、独立生命周期、进行中的过程状态、提交/回滚/恢复/取消/超时/重试、成为新的真相来源
- 判断：只有同时满足“只读、可重算、无独立生命周期、不拥有时序语义”四条，才应该归为 `Select`
- 警告：草稿态、编辑中、乐观更新中、拖拽中、加载中、失败、重试中、分页窗口、本地筛选条件、排序条件、搜索关键字、撤销/重做中的临时状态，通常都不属于 `Select`
- 设立条件：读模型已成为稳定接口，或需要聚合多个 `Store`，或消费方不应直接理解底层 `Store` 结构

## `Loop`

- 定义：重复推进型代码
- 允许：消费时间步长、快照和上下文；逐帧或定步长推进状态；产出状态修改意图或事实
- 禁止：IO、宿主对象、`await`、直接驱动表现、持有权威真相
- 判断：只有真的由 tick 或 fixed-step 驱动，才是 `Loop`

## `Task`

- 定义：有开始和结束的有限流程
- 允许：编排登录、切场景、存档、恢复、匹配、加载、结算提交等有限流程；管理等待、取消、超时、重试、阶段推进；通过显式入口读写 `Store`；通过 `Port` 请求外部能力
- 禁止：tick 推进、宿主回调注册、长期状态真相、持有宿主对象、在内部堆积大量纯计算
- 判断：只要主职责是跨多个步骤把一件事办完，它就属于 `Task`，不管是否使用异步语法

## `Entry`

- 定义：所有外部宿主打进项目内部的入口
- 允许：接收引擎生命周期、输入、碰撞、动画、UI、SDK、平台等回调；把外部输入转换成内部调用；读取稳定只读结果用于入口绑定或转发
- 禁止：复杂规则、长期状态、异步编排、直接 IO、复杂流程、隐藏副作用
- 判断：它的特殊性不在“做了什么”，而在“不是被项目内部普通调用进入的”

## `Output`

- 定义：从内部真相或内部事实出发，生成对外效果请求的形态
- 允许：根据稳定读模型或事实决定是否发出效果请求；协调动画、特效、音效、相机、屏幕反馈、UI 反馈等效果意图；准备效果参数
- 禁止：业务规则、权威状态、直接持有宿主对象、主流程编排、IO 细节
- 判断：它决定“是否发效果、发什么效果”；真正操作宿主对象的部分必须下沉到 `Gateway`

## `Port`

- 定义：能力契约
- 允许：定义项目内部如何请求外部能力；约束上层可用的能力入口
- 禁止：实现、状态、流程、宿主对象
- 判断：它只定义“可以怎样请求能力”，不定义“如何实现”；如果接口已暴露具体引擎、平台或 SDK 类型，就不是干净的 `Port`

## `Gateway`

- 定义：唯一允许直接接触宿主不可见对象的形态
- 允许：封装引擎节点、组件、场景句柄、平台对象、SDK 对象、socket、文件句柄、资源句柄；实现 `Port`；做协议映射、宿主 API 适配、错误映射、编解码
- 禁止：业务决策、有限流程编排、权威状态、tick 推进
- 判断：如果换引擎、换平台、换 SDK、换存储介质，需要主要重写的就是它

## `Boot`

- 定义：装配和初始化入口
- 允许：创建对象图、绑定 `Port -> Gateway`、初始化领域上下文、在启动或重建上下文时完成装配
- 禁止：运行期业务控制、可复用规则、长期状态真相、流程细节
- 判断：它只在启动或重建上下文时有意义；许多 `AppManager` / `SceneManager` 文件，往往应拆成 `Boot + Task + Gateway`

## 命名与落位

推荐把主形态直接编码进文件名后缀：

- `*.schema.*`
- `*.pure.*`
- `*.store.*`
- `*.select.*`
- `*.loop.*`
- `*.task.*`
- `*.entry.*`
- `*.output.*`
- `*.port.*`
- `*.gateway.*`
- `*.boot.*`

示例：

- `DamageFormula.pure.cs`
- `Battle.store.cs`
- `BattleHud.select.cs`
- `BuffTick.loop.cs`
- `Login.task.cs`
- `PlayerInput.entry.cs`
- `HitFlash.output.cs`
- `Audio.port.cs`
- `Audio.gateway.cs`
- `Battle.boot.cs`

命名规则：

- 主文件名表达领域对象、动作或边界对象
- 后缀表达主形态
- 一个文件只能有一个主形态后缀
- 禁止使用不带形态信息的最终归宿命名
- 目录可以按领域聚合，也可以按形态聚合；但不应省略文件级形态标识
