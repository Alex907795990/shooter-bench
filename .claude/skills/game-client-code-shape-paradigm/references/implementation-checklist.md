# 实现前后检查清单

## 何时读取

- 准备开始实现
- 准备宣称“已完成”
- review 时需要快速判断是否已经越界

## 实现前必须输出

开始写代码前，先明确给出以下结论：

```md
- 主形态：
- 命中的判定链：
- 入口来源：
- 是否直接接触宿主不可见对象：
- 是否持有某个作用域内的权威真相：
- 执行形态（同步计算 / 有限流程 / 重复推进）：
- 允许依赖的形态：
- 绝对禁止做的事：
- 将使用的正式通信形态：
- 当前目标文件是否已越界；若越界，拆分方案是什么：
```

如果这 10 项里有任何一项答不稳，不要进入实现。

## 实现后必须逐项自检

实现完成后，逐项回答是/否：

1. 是否新增跨形态混写
2. 是否让同一文件同时承担 `Callback` 与 `Await`
3. 是否让同一文件同时承担 `Tick` 与 IO
4. 是否让宿主不可见对象泄露出 `Gateway`
5. 是否让 `Select` 承载了过程真相
6. 是否让多个 `Store` 同时声称同一份事实
7. 是否让 `Output` 直接依赖 `Store` 内部细节
8. 是否引入了依赖矩阵之外的调用
9. 是否通过共享引用、单例、事件总线隐藏了真实写入路径
10. 是否遗留旧入口、旧实现、双通路逻辑或无形态归宿的新文件

只要任一项答案为“是”，就不能宣称任务完成。

## 一票否决项

以下情形一旦出现，直接视为结构错误：

- 在同一文件中同时承载 `Callback` 与 `Await`
- 在同一文件中同时承载 `Tick` 与 IO
- 在同一文件中同时承载权威真相与对外效果
- 让外部回调直接改写 `Store`
- 在 `Loop` 中出现 `Await`
- 在 `Gateway` 中按业务语义决定主流程分支
- 用 `Select` 承载草稿态、加载态、回滚态、拖拽态等过程真相
- 因为“这是 UI 的东西”就拒绝为其建立独立 `Store`
- 让多个 `Store` 同时声明同一份事实的权威所有权
- 让 `Output` 直接读取并拼装底层 `Store` 的内部细节
- 让宿主不可见对象从 `Gateway` 泄露到其他形态
- 让 `Signal` 代替 `Store` 充当真相源
- 使用 `utils`、`helper`、`service`、`manager`、`common`、`misc` 作为最终归宿以逃避归类
- 使用未类型化的全局事件总线、服务定位器、单例管理器作为隐藏依赖通道

## 完成标准

- 主形态能够一句话说清
- 判定链、依赖矩阵、通信形态三者彼此一致
- 没有命中一票否决项
- 自检 10 项全部为“否”
