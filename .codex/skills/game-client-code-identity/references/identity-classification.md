# 身份判定

## 何时读取

- 需要在实现前判断主身份
- 一个文件看起来同时像 `System` / `UseCase` / `Presenter` / `Adapter`
- 需要决定当前文件是该保留、拆分还是迁移

## 覆盖范围

本范式覆盖游戏客户端中的主要代码形态，包括但不限于：

- 场景内玩法逻辑
- 战斗、数值、Buff、AI、生成、交互
- HUD、菜单、背包、任务、排行榜、世界空间 UI
- 登录、匹配、进房、切场景、存档、恢复
- 网络、资源、平台、文件、存储、音频、特效、动画、相机接入

不覆盖：

- 编辑器工具
- 构建脚本
- 内容生产流水线
- 服务端代码

## 判定原则

- 先判身份，再允许实现
- 身份按“拥有的能力”判定，不按“语义像什么”判定
- 一个文件或模块只能有一个主身份
- 如果同时命中两个主身份，结论是拆分，不是二选一
- 命中更高权限身份后，不允许人为降级归类
- 如果某段逻辑必须突破依赖矩阵，优先判断为身份判错了

## 闭集身份

| 身份 | 一句话定义 |
| --- | --- |
| `Type` | 静态结构和协议 |
| `Rule` | 纯规则、公式、校验、状态迁移 |
| `State` | 权威运行时状态及其受控更新入口 |
| `System` | 逐帧或定步长推进状态的逻辑单元 |
| `UseCase` | 一次性或阶段性流程编排单元 |
| `Query` | 只读派生、展示数据准备、表现参数准备 |
| `Presenter` | 把状态或事件翻译为引擎可执行的表现动作 |
| `Adapter` | 对外部能力的具体接入层 |
| `Bridge` | 引擎托管脚本及最薄的一层桥接代码 |
| `Port` | 能力契约 |
| `Bootstrap` | 组合根和装配入口 |
| `Command` | 请求系统尝试执行某个动作的协议形态，归类为 `Type` |
| `Event` | 表示某件事已经发生的协议形态，归类为 `Type` |

## 身份判定顺序

按权限从高到低判定，首个命中的身份优先：

1. `Bootstrap`：装配对象图、绑定实现、初始化上下文
2. `Bridge`：直接接收引擎生命周期、输入、碰撞、动画、UI 回调
3. `Adapter`：直接调用网络、存储、文件、平台、资源、场景等外部能力
4. `Presenter`：直接驱动动画、特效、音效、相机、屏幕反馈、UI 表现
5. `System`：按帧或定步长推进状态
6. `UseCase`：编排一次性或阶段性流程
7. `State`：持有权威运行时状态
8. `Port`：只声明能力契约
9. `Query`：只做只读派生
10. `Rule`：只做纯规则、公式、校验、状态迁移
11. `Type`：只描述结构和协议

## 以能力判定，而不是以语义判定

- 只要直接接收引擎回调，它就是 `Bridge`，哪怕里面写的是“业务逻辑”
- 只要直接发网络、读文件、接平台 SDK，它就是 `Adapter`
- 只要直接驱动动画、特效、音效、相机、UI 表现，它就是 `Presenter`
- 只要负责持续推进世界演化，它就是 `System`
- 只要持有跨调用持续存在的权威状态，它就是 `State`
- 只要编排“把一件事办完”的流程，它就是 `UseCase`

## 常见拆分信号

- 同一个文件既接收引擎回调又持有权威状态：拆成 `Bridge` + `State`
- 同一个文件既做流程编排又直接发 IO：拆成 `UseCase` + `Port/Adapter`
- 同一个文件既准备展示数据又直接操作 UI 控件或特效：拆成 `Query` + `Presenter`
- 同一个文件既计算规则又直接改状态：拆成 `Rule` + `System` 或 `Rule` + `UseCase`
- 同一个文件既保存真实状态又承载复杂规则：拆成 `State` + `Rule`

## 裁决规则

- 不能把“为了少拆文件”当成降级归类理由
- 不能用 `service`、`manager`、`helper`、`utils` 一类中性词逃避身份判定
- 一旦发现当前实现需要非法依赖，先重判身份，再考虑拆分方案
