# 身份规则与依赖

## 何时读取

- 需要确认某个身份能做什么、绝对不能做什么
- 需要检查依赖是否合法
- 需要判断 `Command / Event / Query / State` 正式通道是否被绕过

## 身份规则

| 身份 | 负责 | 明确禁止 | 关键判断 |
| --- | --- | --- | --- |
| `Type` | 静态数据结构、值对象、枚举、常量、错误码、配置结构、`Command/Event` 载荷 | IO、异步、引擎对象、时间/随机数/环境变量/全局单例、跨调用持续可变状态、流程编排、业务决策 | 只能回答“数据长什么样” |
| `Rule` | 纯规则、校验、公式、决策逻辑、状态迁移、差异计算、归一化、筛选 | IO、异步、引擎对象、时间/随机数/环境读取、单例、全局可变状态、直接改 `State`、直接驱动表现或副作用 | 相同输入必须得到相同输出 |
| `State` | 权威运行时状态及其受控读取、订阅、更新入口 | IO、引擎 API、复杂规则、多步骤流程、展示拼装、保存引擎对象/节点/组件/平台句柄/网络连接对象 | 删掉它就失去业务真相来源 |
| `System` | 消费状态快照、输入快照、时间步长、事件流，推进冷却、Buff、移动、AI、战斗结算、生成调度等持续演化逻辑 | 任何副作用、网络/存储/文件/平台调用、动画/特效/音效/相机驱动、非逐帧流程、把自己当状态容器 | 负责“世界如何继续演化” |
| `UseCase` | 登录、进房、切场景、存档、恢复、匹配、结算提交等非逐帧流程；可调用 `Rule`、通过正式入口读写 `State`、通过 `Port` 使用外部能力；管理取消、超时、重试、幂等 | 逐帧推进、直接依赖具体 `Adapter`、直接驱动表现、成为长期状态真相、承载大段纯规则 | 负责“把一件事办完” |
| `Query` | 从 `State`、配置和只读输入生成 HUD、菜单、背包、排行榜、小地图、提示文案、表现参数等读模型；可排序、分组、聚合、映射、格式化 | 回写状态、业务决策、副作用、异步、依赖具体 UI 控件或场景对象、成为新的真实状态源 | 只回答“现在该怎么看” |
| `Presenter` | 消费 `Query` 结果或 `Event`，驱动动画、特效、音效、相机、屏幕反馈、UI 表现；协调短生命周期的表现动作 | 玩法规则判定、权威状态推进、替代 `System` 做逐帧玩法演化、网络/存档/平台 IO | 删掉它后，玩法真相不应改变 |
| `Adapter` | 实现 `Port`；执行网络、存储、文件、平台、资源、场景等外部调用；协议映射、错误映射、数据编解码、引擎 API 适配 | 业务决策、跨步骤流程编排、直接改 `State`、直接驱动玩法推进、成为大而全 service | 替换外部实现时主要重写它 |
| `Bridge` | 接收生命周期、输入、碰撞、触发器、动画回调、UI 回调、场景事件；把引擎信号转成 `Command`；读取 `Query` 结果用于展示绑定；转发表现触发给 `Presenter` | 核心规则、权威玩法状态、主流程编排、网络/存档/平台 IO、在回调里堆复杂判断、绕过正式通道直接改系统真相 | 它是壳，不是脑 |
| `Port` | 声明外部能力边界，约束上层如何请求外部能力 | 实现逻辑、保存状态、业务规则、依赖具体引擎对象 | 只定义“需要什么能力” |
| `Bootstrap` | 装配对象图、绑定 `Port -> Adapter`、初始化 `State/System/UseCase/Presenter`，在启动、进场景、重建上下文时完成装配 | 可复用业务逻辑、长期业务状态、规则/查询/流程细节、运行期业务决策或流程控制 | 只存在于启动和装配时刻 |

## 依赖矩阵

只允许以下依赖：

- `Type -> 无`
- `Rule -> Type, Rule`
- `Port -> Type`
- `State -> Type, Rule`
- `Query -> Type, Rule, State(只读)`
- `System -> Type, Rule, State`
- `UseCase -> Type, Rule, State, Port`
- `Presenter -> Type, Query, Port`
- `Adapter -> Type, Port`
- `Bridge -> Type, Query, UseCase, Presenter`
- `Bootstrap -> 全部，但仅限装配期`

补充约束：

- `State` 只能被 `System` 和 `UseCase` 通过明确入口更新
- `UseCase` 和 `Presenter` 只能依赖 `Port`，不能依赖具体 `Adapter`
- `System` 不得依赖 `Port`
- `Bridge` 不得直接依赖 `State`、`Rule`、`System`、`Adapter`
- `Presenter` 不得直接依赖 `State`，所需数据必须通过 `Query`

## 正式通信通道

### `Command`

- 表示“请求系统尝试执行某个动作”
- 所有跨身份的业务改动请求，必须显式表现为 `Command`
- `Command` 必须是显式类型，禁止字符串名加动态参数
- 只有 `Bridge` 和 `UseCase` 可以发起业务 `Command`
- 只有 `System` 和 `UseCase` 可以消费会改变业务真相的 `Command`

### `Event`

- 表示“某件事已经发生”
- `Event` 表示事实，不表示命令
- `System` 和 `UseCase` 可以产出 `Event`
- `Presenter` 可以消费 `Event` 驱动表现
- `Bridge` 只能有限消费 `Event` 做引擎侧绑定，不得据此直接改业务状态

### `Query`

- 所有展示数据和只读派生统一通过 `Query` 提供
- 纯读取不走 `Command` 或 `Event`
- `Query` 的输出不得回写为权威状态

### 状态更新

- `State` 只能通过明确入口更新
- `Bridge`、`Presenter`、`Adapter`、`Query`、`Rule` 一律禁止直接改 `State`
- 禁止通过共享引用、可变别名、全局单例隐藏状态改写

## 一票否决项

以下任一情形都直接视为结构错误：

- 在 `System` 中直接播放音效、实例化特效、改动画参数、改相机、发网络、写存档
- 在 `Bridge` 中持有权威玩法状态，或在引擎回调里堆复杂规则和多步骤流程
- 在 `Adapter` 中按业务语义决定主流程分支，或直接修改 `State`
- 在 `Presenter` 中决定伤害、命中、掉落、冷却、胜负等玩法规则
- 在 `Rule` 或 `Query` 中读取引擎时间、随机源、场景节点、组件句柄、全局单例
- 在 `State` 中保存引擎对象引用、节点、组件、平台句柄、网络连接对象
- 在 `UseCase` 中实现逐帧循环、定步长推进或持续 tick
- 让 `UseCase` 或 `Presenter` 直接依赖具体 `Adapter`
- 让 `Bridge` 绕过 `UseCase`、`System`、`Presenter` 直接发 IO、改状态、做主流程编排
- 用 `utils`、`helper`、`service`、`manager`、`common`、`misc` 作为最终归宿逃避归类
- 一个文件或模块同时承担两个以上主身份
- 使用未类型化的全局事件总线、服务定位器、单例管理器作为隐藏依赖通道
- 把 `Query` 的缓存结果当成新的真实状态源
- 让 `Bootstrap` 在运行期继续承担业务决策或流程控制
