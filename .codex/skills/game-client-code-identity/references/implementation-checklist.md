# 实施与审查清单

## 何时读取

- 开始实现前，需要固定回答模板
- 实现完成后，需要判断是否可以宣称完成
- 需要确认文件命名与落位是否符合身份范式

## 实现前必须回答

在写代码前，先明确回答以下问题：

1. 本次改动的主身份是什么
2. 为什么它不是相邻身份
3. 输入是什么
4. 输出是什么
5. 它允许依赖哪些身份
6. 它绝对禁止做哪些事
7. 当前目标文件是否已越界
8. 如果已越界，拆分方案是什么

可直接使用下面的最小模板：

```text
主身份：
为什么不是相邻身份：
输入：
输出：
允许依赖：
绝对禁止：
目标文件是否越界：
拆分方案：
```

## 实现后必须自检

逐项检查以下问题：

1. 是否新增跨身份混写
2. 是否出现不在依赖矩阵内的依赖
3. 是否存在隐式环境读取
4. 是否绕过了 `Command / Event / Query / State` 正式通道
5. 是否把引擎对象引用带入了 `Rule / Query / State`
6. 是否遗留旧入口、旧实现、双通路逻辑
7. 是否留下无身份归宿的新文件

只要任一项答案为“是”，就不能宣称任务完成。

## 命名与落位

命名建议直接编码主身份：

- `PlayerState.state.cs`
- `DamageFormula.rule.cs`
- `BuffTick.system.cs`
- `Login.usecase.cs`
- `InventoryHud.query.cs`
- `HitFlash.presenter.cs`
- `SaveFile.adapter.cs`
- `PlayerInput.bridge.cs`
- `Audio.port.cs`
- `Battle.bootstrap.cs`

命名规则：

- 主文件名表达业务对象或动作
- 后缀表达主身份
- 禁止使用无身份后缀作为最终落位
- 禁止用 `utils`、`helper`、`service`、`manager`、`common`、`misc` 逃避归类

## 完成标准

只有同时满足以下条件，才可以认为改动符合本范式：

- 新增或修改的文件都能稳定归入单一主身份
- 没有突破依赖矩阵
- 正式状态更新与跨身份通信都通过允许的正式通道
- 没有把引擎对象、平台句柄或隐藏环境带入错误身份
- 没有为了兼容旧实现而留下长期双通路
- 没有新增无法归类的兜底文件
