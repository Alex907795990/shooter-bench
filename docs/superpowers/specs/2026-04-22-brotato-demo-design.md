# 土豆兄弟 Demo 首版设计文档

- 日期：2026-04-22
- 状态：已确认，待执行
- 范围：首版单局制垂直切片

## 1. 目标

首版目标是做一个 `Brotato` 风格的单局制 demo，优先复刻玩法结构，而不是完整复刻数值、美术和全部内容量。玩家应能在一局内完成以下闭环：

`角色选择 -> 5 个战斗轮 -> 战斗轮间商店 -> 最终结算`

该版本应满足：

- 玩法结构接近原作，包含移动生存、自动攻击、敌人掉落材料、战斗轮间商店、武器与道具 build
- 数值和 UI 可以近似，但决策节奏必须成立
- 使用占位资源，不依赖正式美术
- 完整单局时长控制在 10-15 分钟

## 2. 首版范围

### 2.1 包含内容

- 1 个可选角色
- 5 个战斗轮
- 6-8 把武器
- 20-30 个道具
- 3-5 种怪物
- 完整的单局流程与结算
- 角色属性体系
- 武器自动攻击与武器槽
- 武器三合一升阶
- 敌人生成、掉落、拾取
- 商店刷新、购买、锁定、卖出、跳过
- 基础 HUD、商店界面、伤害数字、受击反馈

### 2.2 不包含内容

- 解锁系统
- 元进度、天赋树、难度等级
- 多角色
- Boss、精英、宝箱分支
- 手柄、移动端、联机
- 正式美术和正式音频资产
- 工程、元素、召唤等会显著扩大战斗分支的体系

## 3. 玩法还原原则

- 结构 1:1：必须保留“移动生存 + 自动攻击 + 战斗轮间商店 + build 成长”的核心循环
- 规则近似：商店、武器合成、属性修正、刷怪节奏保留原作味道，但允许首版简化数值
- 展示优先：即使资源占位，也必须保证 HUD、掉落、商店、受击反馈清晰可读
- 单局优先：所有成长都在一局内完成，不引入跨局系统

## 4. 单局流程

### 4.1 基础流程

1. 玩家进入角色选择场景
2. 确认角色后创建 run
3. 依次进行 5 个战斗轮
4. 每个战斗轮结束进入商店
5. 第 5 个战斗轮结束后进入结算

### 4.2 结束条件

- 失败：角色生命归零
- 成功：存活完成第 5 个战斗轮

### 4.3 术语定义

- 战斗轮：指两次商店购买之间的一整次战斗，有独立倒计时。首版 demo 包含 5 个战斗轮。
- 刷怪波次：指战斗轮内部的刷怪单元。一个战斗轮内会持续触发很多刷怪波次，每个刷怪波次可以生成一组或多组敌人。
- 原先容易混用的商店称呼不作为正式术语使用，统一称为战斗轮间商店。

## 5. 战斗轮与战斗规则

### 5.1 战斗轮时长

首版战斗轮时长按原作前短后长的结构设计：

- 第 1 个战斗轮：20 秒
- 第 2 个战斗轮：30 秒
- 第 3 个战斗轮：40 秒
- 第 4 个战斗轮：50 秒
- 第 5 个战斗轮：60 秒

### 5.2 战斗轮结束规则

- 倒计时归零立即结束当前战斗轮
- 不要求清场
- 商店前结算本战斗轮材料与统计信息

### 5.3 战斗内成长

首版不做战斗内升级，不做暂停三选一。

敌人只掉落材料货币。玩家成长完全来自：

- 初始角色属性
- 武器购买与升阶
- 道具购买
- 商店中的 build 选择

### 5.4 玩家战斗方式

- 输入：`WASD` 移动
- 攻击：全自动
- 不做鼠标瞄准

## 6. 武器系统

### 6.1 武器槽

- 最多装备 6 把武器
- 武器自动攻击
- 商店可购买新武器或补齐同名武器做升阶

### 6.2 武器内容覆盖

首版 6-8 把武器至少覆盖以下攻击型：

- 近战挥砍
- 近战突刺
- 单发远程
- 高攻速低伤远程
- 散射远程
- 慢速高伤远程

### 6.3 武器关键字段

每把武器至少具备以下数据：

- 攻击间隔
- 基础伤害系数
- 攻击范围或射程
- 命中形态
- 目标选择规则
- 暴击或击退倾向
- 稀有度与价格

### 6.4 升阶规则

- 保留“三合一升阶”
- 同名同阶武器凑满时合成更高阶武器
- 升阶后继承更强数值和更高商店权重价值

## 7. 道具与属性系统

### 7.1 首版属性范围

首版道具与角色修正围绕以下属性展开：

- 伤害
- 攻速
- 暴击
- 生命上限
- 生命回复或吸血
- 移速
- 闪避
- 幸运
- 收获

### 7.2 设计目标

- 道具以被动数值修正为主
- build 至少能形成输出、生存、经济 3 类倾向
- 商店中“补武器体系”和“堆属性道具”必须形成取舍

## 8. 怪物与生成系统

### 8.1 怪物原型

首版 3-5 种怪物原型建议覆盖：

- 追击近战
- 高血慢速坦克
- 远程投射
- 快速骚扰或冲刺型

### 8.2 生成规则

- 每个战斗轮使用数据驱动的刷怪时间表，而不是在系统代码中写死时间点
- 刷怪时间表由多个刷怪波次组成；每个刷怪波次定义出现时间、重复次数、重复间隔、敌人类型、数量范围、出现概率和生成位置规则
- 生成位置规则至少支持随机点、按群出现、地图边缘出现
- 敌人实际出现前必须先创建占位提示标记，延迟后再生成敌人
- 同一刷怪波次内允许有随机数量、随机落点和轻微分批落地延迟，以形成随机感和群体感
- 第 5 个战斗轮通过更高刷怪波次密度、更短重复间隔和混编敌种形成高潮，不做 Boss

### 8.3 原作刷怪机制调研结论

调研来源：

- Brotato Wiki 的敌人页面：<https://brotato.wiki.spellsandguns.com/Enemies>
- Brotato Wiki 的 More Enemies 页面：<https://brotato.wiki.spellsandguns.com/More_Enemies>
- ArosRising 战斗轮与敌人指南：<https://gameplay.tips/guides/brotato-ultimate-guide-to-enemies-and-waves.html>
- DarkTwinge Brotato BalanceMod 的战斗轮刷怪资源：<https://github.com/DarkTwinge/Brotato-BalanceMod/tree/master/waves>
- Steam 社区关于红 X 刷怪占位的讨论：<https://steamcommunity.com/app/1942280/discussions/0/3810656958845428240/>

确认事实：

- 原作刷怪不是单纯的“预算 + 权重 + 曲线”连续随机系统，而是更接近数据驱动的战斗轮刷怪表。
- 每个战斗轮的刷怪表由多个刷怪波次组成。公开 mod 资源中的 `wave_group_data` 字段包含 `spawn_chance`、`spawn_timing`、`repeating`、`repeating_interval`、`reduce_repeating_interval`、`area`、`spawn_edge_of_map`、`wave_units_data`、`is_horde`、`min_wave`、`max_wave` 等。
- 刷怪波次引用一个或多个单位数据。公开 mod 资源中的 `wave_unit_data` 字段包含 `unit_scene`、`min_number`、`max_number`、`spawn_chance`、`additional_min_distance_from_player` 等。
- ArosRising 指南明确说明，指南列出的刷怪时间是红 X 占位标记出现的时间，敌人会在红 X 出现 1 秒后生成。
- 同批大量敌人不会完全同一帧落地，而是会在约半秒内陆续生成。
- 原作敌人的位置规则包括默认随机点、按群出现、地图边缘出现、不能彼此靠太近。按群出现通常在 200-300 范围内，也有更大的组半径。
- 玩家站在红 X 上不会让刷怪丢失，敌人会换到其他位置生成。
- 原作每个战斗轮内场上敌人上限为 100。`+% Enemies` 增加每次刷出的敌人数，但不提高场上上限。

首版采用原则：

- 保留原作的“时间表 + 重复间隔 + 群体出现 + 红 X 延迟生成”手感。
- 不复制完整 20 个战斗轮与全部敌人数值，只为 5 个战斗轮 demo 制作小规模刷怪波次数据。
- 不把刷怪波次时间点写在 `System` 的条件分支中，刷怪波次内容必须来自可读的数据定义。
- 刷怪相关跨帧状态放入 `InstanceContainer`，例如当前战斗轮进度、刷怪波次进度、待生成占位标记、下一次重复触发时间。
- 刷怪业务逻辑放在 `System` 中：一个系统负责到点创建占位标记，另一个系统负责占位延迟结束后提交敌人生成。
- 占位创建和敌人生成都应产生 `Event`，供 Phaser 表现层同步红 X 标记、敌人出生反馈和调试信息。

## 9. 商店系统

### 9.1 商店能力

首版商店支持：

- 购买
- 刷新
- 锁定
- 卖出武器
- 离开商店

### 9.2 商品结构

- 商店默认展示 4 个商品槽
- 商品来自武器池和道具池的加权随机
- 受稀有度和当前局内 build 轻度影响
- 锁定商品可保留到下一次商店

### 9.3 决策目标

商店必须让以下抉择成立：

- 现在补武器，还是补属性
- 现在刷新，还是保留材料
- 现在卖出旧武器，还是保留组合可能性

## 10. 场景与界面

### 10.1 场景拆分

- `Boot`：装配与启动
- `CharacterSelect`：角色确认
- `Battle`：战斗与 HUD
- `Shop`：战斗轮间商店
- `Result`：胜负结算

### 10.2 HUD 最低要求

HUD 必须持续展示：

- 当前生命
- 当前材料
- 当前战斗轮
- 剩余时间
- 武器槽摘要

### 10.3 反馈最低要求

战斗与商店至少需要：

- 伤害数字
- 玩家受击反馈
- 材料掉落与拾取反馈
- 商品卡信息与购买结果反馈

## 11. 完成定义

首版满足以下条件才算完成：

- 能稳定跑完完整 5 个战斗轮流程
- 每个战斗轮结束都能稳定进入商店并返回下一个战斗轮
- 敌人会生成、移动、攻击、掉落材料
- 玩家可拾取材料并在商店消费
- 武器会自动攻击，6 武器槽与三合一升阶可用
- 道具购买后能立刻影响战斗属性
- HUD 和商店信息对首次体验者可读
- 不存在阻断主流程的严重 bug
